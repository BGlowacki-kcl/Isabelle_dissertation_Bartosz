# PIDE Bug Report — xml.scala

Three bugs found, confirmed, fixed, and verified in
`Isabelle2025/src/Pure/PIDE/xml.scala`.

All tests went **FAIL → PASS** after applying the fixes.

---

## Bug 1 — `XML.Output.string(null)` appends the literal string `"null"`

### Location
`xml.scala` — class `Output`, method `string`, line ~195

### What the bug is

```scala
// BUGGY
def string(str: String, permissive: Boolean = false): Unit = {
  if (str == null) { builder ++= str }   // ← builder.append(null) → "null"
  else {
    str foreach { case '<' => … }
  }
}
```

The method contains a defensive null check, but the action taken is wrong:
`builder ++= str` (with `str == null`) calls Java's
`StringBuilder.append(null: String)`, which appends the four ASCII characters
`n`, `u`, `l`, `l`.  The `else` branch that would HTML-escape meaningful
characters is never reached.

### The test

```scala
val result = XML.string_of_tree(XML.Text(null))
result == "null"   // true  → bug confirmed
```

`XML.string_of_tree` traverses the tree and calls `text(s)` → `string(s)` for
each text node.  With `s == null` the buggy branch fires and the serialised
output contains the literal `null`.

### The fix

```scala
// FIXED
if (str == null) { /* ignore null text — must not append the literal "null" */ }
```

Null text is treated as an empty string (no output produced).

### Why the fix helps

The `else` branch already produces the correct output for non-null strings.
For null the correct output is nothing (an empty text node contributes zero
characters).  Skipping the append entirely restores that contract.

### Consequences for the end user

Any XML.Text node whose `content` field is null (possible via Java interop or
an ML prover malfunction) produces corrupt serialised output: the four
characters `null` appear inline in what the frontend sends to the backend or
renders to the user.  This can:

- Corrupt protocol messages between the Isabelle/PIDE session manager and the
  ML prover, causing parse errors or unexpected behaviour in proof state display.
- Insert the literal word "null" into rendered proof output visible in editors
  (jEdit, VS Code), silently misleading the user.
- Break XML-based export of proofs or theories that contain null text nodes.

## Bug 2 — `XML.filter_elements` crashes with `StackOverflowError` on deep trees

### Location
`xml.scala` — function `filter_elements`, inner function `filter`, line ~148

### What the bug is

```scala
// BUGGY — non-tail-recursive
def filter_elements(xml: XML.Body, …): XML.Body = {
  def filter(ts: XML.Body): XML.Body =
    ts flatMap {
      case XML.Elem(markup, body) =>
        …
        else List(XML.Elem(markup, filter(body)))   // ← recursive JVM call
      …
    }
  filter(xml)
}
```

The inner `filter` function calls itself once per nesting level.  For a tree
of depth D it requires D nested JVM stack frames.  With the JVM's default
thread stack of 512 KB this exhausts at roughly 2 000–4 000 levels for this
method signature.  Isabelle proof trees—especially those generated by
automated tactics operating on large terms—can reach this depth.

### The test

```scala
val deepTree = buildDeepTree(5000)   // iterative construction, no JVM overflow
try { XML.filter_elements(List(deepTree)); false }
catch { case _: StackOverflowError => true }   // true → bug confirmed
```

### The fix

Replace the recursive `filter` with an iterative algorithm that uses an
explicit heap-allocated work queue and accumulator stack:

```
Op = DoTree(t) | CloseElem(markup) | CloseWrapped(markup, body1)
work: List[Op]         — pending operations (heap, not JVM stack)
accs: List[ListBuffer] — output accumulator stack

For each DoTree(Elem(markup, body)):
  → push CloseElem(markup) and DoTree for each child into work
  → push fresh accumulator for children's output
For DoTree(Text / leaf): add to current accumulator
For CloseElem(markup):
  → pop children accumulator, wrap in Elem, add to parent accumulator
```

The algorithm is semantically identical to the recursive version; only the
execution mechanism changes (heap stack vs JVM call stack).

### Why the fix helps

Every recursive JVM call is replaced by a heap allocation.  Heap memory is
orders of magnitude larger than the thread stack; a tree of depth 100 000
would require only ~100 000 entries on the work list, negligible memory.

### Consequences for the end user

`filter_elements` is called from multiple PIDE components to strip internal
markup (e.g. removing proof-internal annotations before display).  A crash
inside `filter_elements` propagates `StackOverflowError` to:

- The session manager, which may kill the entire Isabelle session process,
  forcing the user to restart and lose unsaved proof state.
- The document rendering pipeline in jEdit / VS Code, which may freeze or
  crash the editor plugin.
- Proof export functions that filter markup from terms before writing them to
  files or databases.

The failure is silent from the user's perspective: the editor simply becomes
unresponsive or disconnects without explanation.

## Bug 3 — `XML.Cache.tree()` crashes with `StackOverflowError` on deep trees

### Location
`xml.scala` — class `XML.Cache`, methods `cache_tree` / `cache_body` (~line 279)
and `XML.Elem.hashCode()` (~line 28)

### What the bug is (two interacting causes)

**Cause A — recursive `hashCode` computation in `XML.Elem`:**

```scala
// BUGGY
case class Elem(markup: Markup, body: Body) extends Tree {
  private lazy val hash: Int = (markup, body).hashCode()
  //                                          ↑ calls body.hashCode()
  //                                            → child.hashCode()
  //                                              → grandchild.hashCode()
  //                                                → … (D levels deep)
  override def hashCode(): Int = hash
}
```

`(markup, body).hashCode()` delegates to `List[Tree].hashCode()`, which
recursively calls `hashCode()` on each element.  For a tree of depth D this
creates a call chain D frames deep.  The lazy `hash` field is evaluated on
first use—the first time the tree is inserted into or looked up in any hash
collection (including Java's `WeakHashMap` used by the cache).

**Cause B — mutually recursive `cache_tree` / `cache_body`:**

```scala
// BUGGY
protected def cache_tree(x: XML.Tree): XML.Tree =
  lookup(x) match {
    case None =>
      x match {
        case Elem(markup, body) =>
          store(Elem(cache_markup(markup), cache_body(body)))  // ← calls cache_body
```

```scala
protected def cache_body(x: XML.Body): XML.Body =
  lookup(x) match {
    case None => x.map(cache_tree)   // ← calls cache_tree per child
  }
```

Even if Cause A is fixed (hashCode becomes shallow), the mutual recursion
between `cache_tree` and `cache_body` creates one JVM frame per nesting level.

### The test

```scala
val deepTree = buildDeepTree(5000)
val cache    = XML.Cache.make()
try { cache.tree(deepTree); false }
catch { case _: StackOverflowError => true }   // true → bug confirmed
```

### The fix (two parts)

**Fix A — shallow `hashCode` for `XML.Elem`:**

```scala
// FIXED
private lazy val hash: Int = markup.hashCode() * 31 + body.length
```

The hash now depends only on the element's immediate markup and the number of
direct children.  This is O(1) regardless of tree depth.  Hash collisions are
more frequent (two elements with the same markup and same number of children
collide) but correctness is maintained: `equals` still performs full structural
comparison when needed.

**Fix B — iterative `cache_tree`:**

```scala
// FIXED — iterative using explicit work queue and accumulator stack
protected def cache_tree(x: XML.Tree): XML.Tree = {
  // Op = DoTree(t) | CloseElem(cached_markup)
  // work: List[Op], accs: List[ListBuffer[Tree]]
  …while loop, no recursive calls…
  accs.head.head
}
```

The iterative version replaces the JVM call stack with a heap-allocated work
queue (same pattern as the `filter_elements` fix).

### Why the fix helps

Fix A eliminates the overflow that occurs on the very first `lookup(root)` call
(before any iteration).  Fix B eliminates the overflow that would occur in the
recursive descent even if Cause A were fixed independently.  Together they make
`cache.tree(deepTree)` run in O(N) heap operations with O(depth) heap memory
but O(1) JVM stack depth.

### Consequences for the end user

`XML.Cache.tree()` is called pervasively throughout the PIDE session:

- **Every markup tree received from the ML prover** is passed through the cache
  for structural sharing (interning).  A large Isabelle proof (automated tactic
  proof, large inductive definition, etc.) can produce terms with deep nesting.
- A `StackOverflowError` thrown inside the `synchronized` cache block propagates
  to the session manager's consumer thread.  Because this thread has no recovery
  for `Error`-level exceptions, it typically terminates and the Isabelle session
  becomes permanently unavailable until the user restarts it.
- In interactive proofs this means losing all unsaved proof state.  In batch
  processing it means silently failing to process one or more theories, possibly
  without any error visible to the user.
